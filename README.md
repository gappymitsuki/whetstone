# Whetstone — Demo MVP

🌐 **ライブデモ：https://whetstone-gamma.vercel.app**

**専門家の判断を「砥石」に当てて、AIエージェントを研ぎ上げるランタイム層。**

このデモが証明するただ一つのこと：
**専門家が数件ジャッジしただけで、AIエージェントのエスカレーション率が目に見えて下がる。**

ドメインは「広告コピーのブランドガイドライン適合判定」（架空の文房具ブランド Nagomi）。

---

## コアループ

```
エージェントがコピーを判定（適合/違反/要確認 + 確信度 + 理由）
  → 確信度が閾値未満 or 要確認 → 専門家レビュー・キューへエスカレ
  → 専門家が 適合/違反 + 一言理由 を入力
  → その判断を「汎用ルール」に変換して判断ルール集へ蓄積
  → 次バッチでは拡張ルール集を参照 → 同種ケースが高確信で自走 → エスカレ率が下がる
```

エスカレ率が下がるのは演出ではなく、**ルールが増えて確信度が物理的に閾値を超えるから**。

### 環境変数（`.env.local`）

| 変数 | 説明 |
|---|---|
| `OPENAI_API_KEY` | OpenAI API キー。**サーバーサイドのみ**で使用（クライアントには晒さない）。未設定でも動く（自動でオフライン判定にフォールバック）。 |
| `NEXT_PUBLIC_OFFLINE_DEMO` | `1` で**オフラインデモモード**。LLM を一切呼ばず、決定的なローカル判定で動作。ネット不通でも通しで動く。**本番ピッチではこれを推奨**。 |

> モデル名・閾値などの定数は `lib/config.ts` に集約。差し替えはここだけ。

## デモ手順（壇上 2〜3分・ノー編集で再現可能）

1. **リセット** を押す（学習前の状態へ）。
2. **バッチ実行**（バッチ1）。グレー類型5件がエスカレし、ダッシュボードの**エスカレ率が高め（50%）**に出る。
3. 右下の**レビュー・キュー**で、グレーケースを3件ほど「違反」＋一言理由で**確定**。
   - キューは論点ごとに **法務・コンプラ / ブランド審査** の専門家レーンへ**自動ルーティング**済み（スマートエスカレーション）。各ケースに「ルーター判定：◯◯の論点」が表示される。
   - ルール集に汎用ルールが蓄積され（どの専門家レーン由来かもタグ表示）、キューから消える。
4. **バッチ実行**（バッチ2）。同じグレー類型が今度は**高確信で自走**（依拠ルールIDが表示される）。
5. ダッシュボードの**エスカレ率の折れ線がガクッと下降**（50% → 20%）。正答率は高いまま。

> 推奨：本番では `NEXT_PUBLIC_OFFLINE_DEMO=1` で起動し、ネット環境に依存せず同じ流れを確実に再演する。

## 事故耐性

- **API 失敗時**：リトライ上限後も失敗したら、サーバー側で各シードケースの事前計算済みオフライン判定にフォールバック（`app/api/judge/route.ts`）。クライアント側にも二重のフォールバックがあり、**アプリは絶対に落とさない**。
- **オフラインデモモード**：`NEXT_PUBLIC_OFFLINE_DEMO=1` で LLM を完全にバイパス。ネット不通でもデモが成立。
- **再現性**：temperature 低め。**リセット**で完全に学習前へ戻せる。

## ドメイン差し替え

題材を別ドメインに変える場合、編集するのは以下の3ファイルだけ：

- `data/guideline.ts` — 審査ガイドライン本文
- `data/cases.ts` — シードケース（バッチ別）と類型情報 `GREY_TYPE_INFO`
- `data/experts.ts` — 専門家レーンの定義と、論点→専門家のルーティング規則

ロジック（`lib/engine.ts`）にはドメイン知識をハードコードしていない。

## ディレクトリ構成

```
app/
  page.tsx                     # オペレーター・コンソール（メイン。レビューUI/ルール集も同居）
  api/judge/route.ts           # 判定LLM呼び出し（+ オフラインフォールバック）
  api/generalize-rule/route.ts # 専門家判断→汎用ルール変換
lib/
  config.ts    types.ts    engine.ts    prompts.ts    openai.ts
data/
  guideline.ts   cases.ts   experts.ts   # experts: 専門家レーン定義＋ルーティング規則
components/
  CaseList.tsx  ConfidenceBar.tsx  Dashboard.tsx  ReviewQueue.tsx  RuleBook.tsx
```

永続化は無し（すべてメモリ上の state。リロードで初期化）。Vercel デプロイ想定。

## スコープ外（作っていない）

認証・課金・DB・外部永続化・統計的較正、
そして**蓄積判断の外部エクスポート／資格・認証レイヤー**（意図的にスコープ外。UIにも出さない）。

> 専門家レーンへの振り分け（スマートエスカレーション）は、競合優位の「ルーティング構造」を示すデモ用の最小実装。
> 実運用で必要な負荷分散・専門家のスキルマッチ最適化・SLA 等までは作り込んでいない。
